# Clean Architecture: краткое напоминание

## Что такое Clean Architecture

Clean Architecture — это архитектурный подход, предложенный Робертом Мартином (Uncle Bob), главная цель которого — сделать систему независимой от деталей реализации.

Идея заключается в том, что бизнес-логика должна существовать сама по себе и не зависеть от базы данных, веб-фреймворка, UI, очередей сообщений или других технологий. Все эти вещи являются лишь инструментами, которые могут изменяться без существенного влияния на бизнес-правила.

В центре архитектуры находится предметная область (Domain), а остальные слои строятся вокруг неё. При этом зависимости всегда направлены внутрь — внешние слои знают о внутренних, но внутренние ничего не знают о внешних.

---

# Основные принципы Clean Architecture

## 1. Правило зависимостей (Dependency Rule)

Это главный принцип архитектуры.

Исходный код может зависеть только от слоёв, расположенных ближе к центру. Внутренние слои никогда не должны зависеть от внешних.

Например:

* Domain ничего не знает о базе данных;
* Domain ничего не знает о HTTP;
* Domain ничего не знает о Entity Framework, Hibernate или любом другом ORM.

Если завтра заменить PostgreSQL на MongoDB или REST API на gRPC, бизнес-логика останется неизменной.

---

## 2. Разделение ответственности

Каждый слой отвечает только за свою область задач.

Это делает систему проще для понимания, тестирования и сопровождения.

---

## 3. Инверсия зависимостей

Внутренние слои определяют необходимые им интерфейсы, а внешние предоставляют их реализацию.

Получается, что бизнес-логика управляет инфраструктурой, а не наоборот.

---

# Слои Clean Architecture

## Domain

Это ядро системы.

Здесь располагаются:

* сущности (Entities);
* value objects;
* бизнес-правила;
* доменные сервисы;
* доменные события (при необходимости).

Domain ничего не знает о том, как данные сохраняются, как выглядит пользовательский интерфейс или через какой протокол приходит запрос.

Он отвечает только на вопрос:

> **Как работает бизнес?**

Например:

В банковском приложении именно Domain определяет:

* что такое счёт;
* как выполняется перевод;
* какие ограничения существуют;
* когда операция считается допустимой.

---

## Application

Application описывает сценарии использования системы (Use Cases).

Он отвечает на вопрос:

> **Что необходимо сделать для выполнения конкретной операции?**

Например:

"Создать заказ"

Application:

* получает команду;
* валидирует её;
* вызывает методы Domain;
* сохраняет изменения;
* публикует события;
* возвращает результат.

Важно понимать, что Application не содержит сложных бизнес-правил. Он лишь координирует выполнение операций.

Именно здесь обычно располагаются:

* Commands;
* Queries;
* Command Handlers;
* Query Handlers;
* DTO;
* интерфейсы репозиториев;
* интерфейсы внешних сервисов.

---

## Interface (Presentation)

Этот слой отвечает за взаимодействие с внешним миром.

Это могут быть:

* REST API;
* GraphQL;
* gRPC;
* консольное приложение;
* веб-интерфейс;
* мобильное приложение.

Его задача — преобразовать внешний запрос во внутреннюю команду Application и вернуть ответ в удобном для клиента формате.

Этот слой не должен содержать бизнес-логику.

---

## Infrastructure

Infrastructure содержит конкретные технические реализации.

Здесь располагаются:

* работа с базой данных;
* файловая система;
* очереди сообщений;
* внешние API;
* отправка электронной почты;
* кэширование;
* логирование;
* реализации репозиториев.

Именно этот слой зависит от технологий и библиотек.

Если необходимо заменить одну СУБД на другую, изменения чаще всего ограничиваются Infrastructure.

---

# Репозиторий в Infrastructure

Особое внимание стоит уделить репозиториям.

Репозиторий — это реализация интерфейса, объявленного в Application (или иногда в Domain).

Например, Application может определить интерфейс:

* IUserRepository
* IOrderRepository

Но ничего не знать о том, где физически хранятся данные.

Infrastructure реализует этот интерфейс:

* через PostgreSQL;
* SQL Server;
* MongoDB;
* Redis;
* внешний REST API;
* любой другой источник данных.

Для бизнес-логики все эти варианты выглядят одинаково.

Она просто вызывает:

```
orderRepository.GetById(id)
orderRepository.Save(order)
```

и не знает, каким образом данные были получены.

Это позволяет:

* легко менять источник данных;
* писать модульные тесты с использованием моков;
* изолировать бизнес-логику от инфраструктурных деталей.

Важно помнить, что репозиторий не должен содержать бизнес-логику. Его задача — предоставить доступ к данным и сохранить изменения.

---

# Итог

Clean Architecture позволяет разделить систему на независимые слои, где каждый отвечает только за свою область ответственности.

Ключевые идеи можно свести к четырём тезисам:

* бизнес-логика находится в центре системы;
* зависимости всегда направлены внутрь;
* инфраструктура является заменяемой деталью реализации;
* репозитории скрывают способ хранения данных и предоставляют единый интерфейс доступа к информации.

Такое разделение делает систему проще в сопровождении, тестировании и развитии, а также позволяет менять технологии без переписывания бизнес-логики.


---

Общий репозиторий нужен не для того, чтобы не писать CRUD. CRUD — это лишь несколько методов. Его главная ценность в том, что он устраняет необходимость создавать отдельный метод под каждую комбинацию условий поиска. Вместо роста количества методов растут только параметры фильтрации.

```python
await repo.list_active_admins_created_after(
    date,
)

await repo.list_users_by_departments(
    departments,
)

await repo.list_users_by_roles(
    roles,
)

await repo.list_users_by_name_prefix(
    prefix,
)
```

```python
await repo.list(
    is_active=True,
    role="admin",
    created_at__gte=date,
)

await repo.list(
    department__in=departments,
)

await repo.list(
    role__in=roles,
)

await repo.list(
    username__ilike=f"{prefix}%",
)
```

```
Это хорошо ложится в рассказ

Я бы построил повествование так:

Без общего репозитория

"Каждый репозиторий содержит одинаковый CRUD."

↓

Добавили общий репозиторий

"CRUD переехал в базовый класс."

↓

Добавили универсальную фильтрацию

"Исчезли десятки методов find_by_*."

↓

Добавили Generic

"При этом мы не потеряли типизацию."

И закончить можно примерно такой фразой:

Несмотря на то, что реализация стала универсальной, с точки зрения разработчика ничего не изменилось. user_repo.get() по-прежнему возвращает User, а order_repo.list() — list[Order]. Универсальность осталась внутри репозитория, а снаружи сохранился строгий контракт и полная поддержка статической типизации.

Мне кажется, именно это будет самым сильным аргументом для Python-разработчиков: универсальность обычно ассоциируется с потерей типов (Any), а здесь вы показываете, что благодаря Generic[T] удалось получить и гибкость, и строгую типизацию одновременно.
```
